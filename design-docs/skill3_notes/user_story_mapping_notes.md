# User Story Mapping

**Overview**
User story mapping is a lean UX mapping method. It uses sticky notes to outline the steps and interactions users will go through within a digital product. Story maps are visualized from the product's perspective and consist of three different levels of user actions.

## Three Levels of User Actions

### 1. Activities (Level 1)
- **Definition:** Represent the core tasks users aim to complete in a product or feature.
- **Characteristics:**
  - Usually only a few high-level activities are included.
  - Displayed horizontally in chronological order.
  - Can be mapped in parallel if designing for multiple users and paths.
- **Visual Cue:** Use the same color sticky note for all activities.
- **Example:** In a mobile banking app, activities could be "checking an account balance" and "depositing a check".

### 2. Steps (Level 2)
- **Definition:** Specific sub-tasks users go through to complete an activity.
- **Characteristics:**
  - Sit directly underneath their corresponding activities.
  - Displayed in chronological order on the map.
- **Visual Cue:** Use a different color sticky note to visually differentiate steps from activities.
- **Example:** For the activity "depositing a check", steps would be:
  1. Entering the mobile deposit details.
  2. Photographing the check.
  3. Submitting the check.
  4. Confirming the deposit.

### 3. Details (Level 3)
- **Definition:** Granular interactions necessary for users to complete the step above.
- **Visual Cue:** Use a third color for sticky notes to keep the map organized and distinct.
- **Example:** For a "login" step, details might be "entering a username/email address" and "entering a password" (as two separate details).

## Agile Release Planning and Adaptation

- **Release Lines:** Once the map is complete, agile teams draw horizontal release lines.
  - **In-Release:** Details included in a specific release are moved above the line.
  - **Deferred:** Details moved below the line become candidates for future sprints.
- **Adaptation:** As teams learn and prioritize over time, deferred details, new activities, steps, and new details can be swapped and adjusted.
- **Benefits over Traditional Methods:** It is much easier to swap out sticky notes during product planning and ideation than it is to revise hefty requirement documents or rewrite code.

## Key Benefits of User Story Maps
1. **Define Scope:** Helps agile teams define what to build.
2. **Preserve Visibility:** Shows how everything fits together holistically.
3. **Facilitate Alignment:** Enables user-centered conversations, team collaboration, and feature prioritization.
4. **Guide Development:** Aligns and guides iterative product development.
