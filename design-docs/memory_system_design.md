# Autonomous AI Harness: Memory System Design

This document details the memory architecture for the autonomous AI agent harness, designed specifically for a solo developer running rapid MVP cycles using the 4-skill architecture.

The memory system utilizes LangChain's **Deep Agents** framework, specifically leveraging the `CompositeBackend` to orchestrate both ephemeral state for active runs and durable storage for long-term learning and history.

## 1. Scope & Identity Configuration

Since the harness is designed for a solo developer, the memory scope will be **Agent-Scoped** (or effectively a static user-scope). We will define a static namespace so that every invocation of the harness accesses the exact same persistent memory pool.

**Implementation Details:**
- **Namespace Factory:** `lambda rt: ("solo-dev-harness",)`
- This ensures all historical data, logs, and preferences are accumulated in one centralized place, building a unified identity and historical context for the agent.

## 2. Backend Routing (`CompositeBackend`)

The memory system uses a `CompositeBackend` to intelligently route filesystem operations based on the path. This cleanly separates the transient data needed to generate a single MVP from the long-term knowledge the agent accumulates.

### Architecture
```python
from deepagents.backends import CompositeBackend, StateBackend, StoreBackend

backend = CompositeBackend(
    default=StateBackend(), # Ephemeral storage for active runs
    routes={
        "/memories/": StoreBackend(
            namespace=lambda rt: ("solo-dev-harness",)
        ),
    }
)
```

## 3. Storage Strategy: Ephemeral vs. Persistent

### Ephemeral Storage (Routed to `default: StateBackend()`)
Data that only matters for the *current* execution of the AI UX Process. This data is wiped/inaccessible after the thread completes, preventing clutter in the long-term database.

*   **/workspace/current_problem.md**: Scratchpad for `problem-and-hypothesis-analyzer` to draft pain points.
*   **/workspace/risk_reward_scratchpad.md**: Intermediate calculations and evaluations by the `risk-reward-evaluator`.
*   **/workspace/mvp_core_draft.md**: The working draft of user stories and minimal flows created by `mvp-core-architect`.
*   **/workspace/final_prompt.md**: The final output generated by `ai-coder-prompt-engineer` before it is handed off to the coding agent.

### Persistent Storage (Routed to `/memories/`: `StoreBackend()`)
Data that needs to survive across multiple executions of the harness to build historical context, log past actions, and inform future decisions.

*   **/memories/current_project_context.md**:
    *   *What it is:* The finalized output of the `problem-and-hypothesis-analyzer` (Problem statement, Value-Proposition Hypothesis, Solution Hypothesis).
    *   *How it's used:* Written by the analyzer skill. Read by the evaluator, architect, and prompt-engineer skills to ensure they are aligned with the core hypotheses throughout the current run. Overwritten at the start of a *new* MVP project.
*   **/memories/hypotheses_log.md** (or `/memories/hypotheses_log/` directory):
    *   *What it is:* An append-only log (Episodic memory) of all hypotheses ever generated and tested.
    *   *How it's used:* The `risk-reward-evaluator` and `problem-and-hypothesis-analyzer` append to this file. Before generating new hypotheses, the analyzer can read this log to avoid repeating past failed ideas or to iterate on past successful ones.
*   **/memories/harness_preferences.md**:
    *   *What it is:* Semantic memory containing preferences for the solo developer (e.g., preferred tech stack, preferred tone, specific edge cases to always look out for).
    *   *How it's used:* Read by all skills to align the generated outputs with the developer's preferences.

## 4. Skill Access and Integration

All four skills (`problem-and-hypothesis-analyzer`, `risk-reward-evaluator`, `mvp-core-architect`, `ai-coder-prompt-engineer`) will be granted **Read/Write access** to the memory.

*   **Read Access:** Skills will natively use tools like `ls`, `read_file`, `grep`, and `glob` to read the context and history.
*   **Write Access:** Skills will use the `write_file` and `edit_file` tools to update the persistent state.
    *   *Example:* Once the `problem-and-hypothesis-analyzer` finalizes the hypotheses, it executes `write_file("/memories/current_project_context.md", <content>)`.
    *   *Example:* The analyzer also executes `edit_file` to append the new hypotheses to the bottom of `/memories/hypotheses_log.md`.

*Note on Skill Definitions:* As requested, the `SKILL.md` definitions are hardcoded and will reside on the standard local filesystem, loaded at initialization via the `skills=["/path/to/local/skills/"]` argument, separate from the agent's dynamic memory store.
